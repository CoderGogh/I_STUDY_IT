## 1. 살충제 패러독스

> 동일한 테스트 케이스에 의한 **반복적 테스트는 새로운 버그를 찾지 못한다**는 테스트의 원리

---

## 2. 데이터 마이닝

> 대규모로 저장된 데이터 안에서 체계적이고 자동적으로 **통계적 규칙이나 패턴을 찾아내는 기술**

- 광석에서 금을 캐내는 것에 비유
- 수많은 데이터에서 **가치 있는 유용한 정보를 찾아내는 것**

---

## 3. 프로토콜의 3요소

> 프로토콜: 복수의 컴퓨터 사이에서 데이터 통신을 원활하게 하기 위해 필요한 **통신 규약** (예: TCP/IP)

| 요소 | 설명 |
|------|------|
| **구문 (Syntax)** | 시스템 간 정보 전송을 위한 데이터 형식, 코딩, 신호 레벨 등을 규정 |
| **의미 (Semantic)** | 시스템 간 정보 전송을 위한 제어 정보로 조정과 에러 처리를 위한 규정 |
| **타이밍 (Timing)** | 시스템 간 정보 전송을 위한 속도 조절과 순서 관리 규정 |

## 4. XML / JSON

### XML
- 송·수신 시스템 간 데이터 연계의 편의성을 위해 전송되는 **데이터 구조를 동일한 형태로 정의**

### JSON
- 비동기 브라우저·서버 통신(AJAX)을 위해 **'속성-값 쌍', '키-값 쌍'** 으로 이루어진 데이터 오브젝트를 전달하기 위한 개방형 표준 포맷
- 인간이 읽을 수 있는 텍스트 사용

## 5. 스케줄링 알고리즘

### 비선점형 (Non-Preemptive)

> **하나의 프로세스가 끝나지 않으면 다른 프로세스는 CPU를 사용할 수 없음**

| 유형 | 설명 |
|------|------|
| **우선순위 (Priority)** | 프로세스별로 우선순위가 주어지고, 우선순위에 따라 CPU를 할당 |
| **기한부 (Deadline)** | 작업들이 명시된 시간이나 기한 내에 완료되도록 계획 |
| **FCFS** (First Come First Service) | 대기 큐에 도착한 순서에 따라 CPU를 할당 |
| **SJF** (Shortest Job First) | 도착 시점에서 가장 짧은 서비스 기간을 갖는 프로세스를 종료까지 점유 |
| **HRN** (Highest Response Ratio Next) | 현재 응답률이 가장 높은 프로세스 선택 / **응답률 = (대기시간 + 서비스시간) / 서비스시간** |

### 선점형 (Preemptive)

> **하나의 프로세스가 다른 프로세스 대신에 CPU를 차지할 수 있음**

| 유형 | 설명 |
|------|------|
| **RR** (Round Robin) | 시분할 시스템 사용 / FCFS와 유사하나 제한 시간 초과 시 다음 프로세스에 자원 할당 |
| **SRT** (Shortest Remaining Time) | SJF 기법을 선점형으로 변환 |
| **선점 우선순위** | 비선점 우선순위를 선점형으로 변환 |
| **다단계 큐** | 우선순위에 따라 시스템/대화형/일괄처리 프로세스로 나누어 상위·중위·하위 큐로 배치 |
| **다단계 피드백 큐** | 다단계 큐의 단점 보완 / 큐마다 timeout 설정, 초과 시 우선순위 낮은 다음 단계 큐로 이동 |

## 6. 트랜잭션 특성 (ACID)

> 트랜잭션: 데이터베이스 상태를 변화시키는 하나의 논리적 기능을 수행하기 위한 작업 단위

| 특성 | 영문 | 설명 |
|------|------|------|
| **원자성** | Atomicity | 분해가 불가능한 작업의 최소 단위 (All or Nothing) |
| **일관성** | Consistency | 트랜잭션 성공 후 항상 일관된 데이터베이스 상태를 보존 |
| **격리성/독립성** | Isolation | 실행 중인 트랜잭션의 중간 결과를 다른 트랜잭션이 접근 불가 |
| **영속성/지속성** | Durability | 완료된 트랜잭션 결과는 영구적으로 데이터베이스에 저장 |

---

## 7. TCL(Transaction Control Language) 명령어

| 명령어 | 설명 |
|--------|------|
| **COMMIT** | 트랜잭션 확정 / 메모리에 영구적으로 저장 |
| **ROLLBACK** | 트랜잭션 취소 / 내역을 저장 무효화 |
| **CHECKPOINT** | 저장 시기 설정 / 롤백을 위한 시점을 지정 |

## 8. 랜드 어택

> 출발지(Source) IP와 목적지(Destination) IP를 **같은 패킷 주소**로 만들어 보냄으로써, 수신자가 자기 자신에게 응답을 보내게 하여 **시스템의 가용성을 침해**하는 공격 기법

## 9. 해시 암호화 알고리즘

| 알고리즘 | 설명 |
|----------|------|
| **MD5** | 512bit 입력 메시지 블록 처리 → **128bit** 해시값 생성 |
| **SHA-1** | **160bit** 해시값 생성 |
| **SHA-256/384/512** | SHA 알고리즘의 한 종류 / **256bit** 해시값 생성 |
| **HAS-160** | 국내 표준 서명 알고리즘(KCDSA)을 위해 개발된 해시 함수 |

## 10. 결합도 유형

> 결합도: 다른 모듈과의 **의존성 정도** / 낮을수록 좋은 품질

| 결합도 | 설명 |
|--------|------|
| **내용 결합도** (Content) | 다른 모듈 내부의 변수나 기능을 직접 사용 |
| **공통 결합도** (Common) | 모듈 밖에 선언된 전역 변수를 참조 및 갱신 |
| **외부 결합도** (External) | 외부 데이터 포맷, 통신 프로토콜, 디바이스 인터페이스를 공유 |
| **제어 결합도** (Control) | 처리 방법을 지시하는 제어 요소가 전달됨 |
| **스탬프 결합도** (Stamp) | 배열, 객체, 구조 등이 인터페이스로 전달 |
| **자료 결합도** (Data) | 파라미터를 통해서만 모듈 간 상호작용 (가장 좋음) |

**결합도 순서 (높음 → 낮음 = 나쁨 → 좋음)**

```
내용 → 공통 → 외부 → 제어 → 스탬프 → 자료
```

## 11. 응집도 유형

> 응집도: 모듈 내부 요소들이 **하나의 책임/목적을 위해 연결된 정도** / 높을수록 좋은 품질

| 응집도 | 설명 |
|--------|------|
| **우연적 응집도** (Coincidental) | 모듈 내 각 구성요소가 연관이 없음 |
| **논리적 응집도** (Logical) | 유사한 성격이나 특정 형태로 분류된 요소들이 한 모듈에서 처리 |
| **시간적 응집도** (Temporal) | 특정 시간에 처리되어야 하는 활동들을 한 모듈에서 처리 |
| **절차적 응집도** (Procedural) | 모듈 내 구성요소들이 기능을 순차적으로 수행 |
| **교환적 응집도** (Communication) | 동일한 입출력을 사용하여 다른 기능을 수행하는 활동들이 모여 있음 |
| **순차적 응집도** (Sequential) | 한 활동의 출력값을 다른 활동이 사용 |
| **기능적 응집도** (Functional) | 모든 기능이 단일한 목적을 위해 수행 (가장 좋음) |

**응집도 순서 (낮음 → 높음 = 나쁨 → 좋음)**

```
우연적 → 논리적 → 시간적 → 절차적 → 통신적 → 순차적 → 기능적
```

## 13. 반정규화의 주요 기법

> 반정규화: 성능 향상·개발 편의성을 위해 **정규화된 데이터 모델을 의도적으로 통합·중복·분리**하는 행위

| 기법 | 설명 |
|------|------|
| **테이블 병합** | 1:1, 1:M 관계를 통합하여 조인 횟수를 줄여 성능 향상 |
| **테이블 분할** | 수직 또는 수평으로 분할 (파티셔닝) |
| **중복 테이블 추가** | 집계 함수(GROUP BY, SUM 등) 사용 시 별도 통계 테이블을 추가 |
| **컬럼 중복화** | 조인 성능 향상을 위해 중복 허용 |
| **중복 관계 추가** | 여러 경로의 조인 시 성능 저하 예방을 위해 추가적 관계를 맺음 |

## 14. OSI 7계층

> OSI 7계층: ISO에서 개발한 **컴퓨터 네트워크 프로토콜 디자인과 통신을 계층으로 나누어 설명한 개방형 시스템 상호 연결 모델**

| 계층 | 이름 | 설명 |
|------|------|------|
| 7 | **응용 계층** (Application) | 사용자와 네트워크 간 응용서비스 연결, 데이터 생성 |
| 6 | **표현 계층** (Presentation) | 데이터 형식 설정, 부호 교환, 암·복호화 |
| 5 | **세션 계층** (Session) | 송수신 간의 논리적 연결 / 연결 접속, 동기 제어 |
| 4 | **전송 계층** (Transport) | 신뢰성 있는 통신 보장 / 데이터 분할·재조립, 흐름·오류·혼잡 제어 |
| 3 | **네트워크 계층** (Network) | 단말기 간 데이터 전송을 위한 최적 경로 제공 |
| 2 | **데이터링크 계층** (DataLink) | 인접 시스템 간 데이터 전송, 전송 오류 제어 / 동기화, 흐름·회선 제어 |
| 1 | **물리 계층** (Physical) | 0과 1의 비트 정보를 회선에 보내기 위한 전기적 신호 변환 |

## 15. 애플리케이션 성능 측정 지표

| 지표 | 설명 |
|------|------|
| **처리량** (Throughput) | 주어진 시간에 처리할 수 있는 **트랜잭션의 수** |
| **응답시간** (Response Time) | 사용자 입력 종료 후 응답 출력이 시작될 때까지의 **시간** |
| **경과시간** (Turnaround Time) | 요구 입력 시점부터 결과 출력 완료까지 **걸리는 총 시간** |
| **자원 사용률** (Resource Usage) | 트랜잭션 처리 중 사용하는 **CPU·메모리·네트워크 사용량** |

## 16. 비즈니스 연속성 계획(BCP) 주요 용어

> BCP: 각종 재해·재난 발생에 대비하여 **핵심 업무 기능 수행의 연속성을 유지**하는 방법론

| 용어 | 설명 |
|------|------|
| **BIA** (Business Impact Analysis) | 장애·재해로 인한 시간 흐름에 따른 영향도 및 손실 평가 분석 |
| **RTO** (Recovery Time Objective) | 업무 중단 시점부터 **업무가 복구되어 다시 가동**될 때까지의 시간 |
| **RPO** (Recovery Point Objective) | 업무 중단 시점부터 데이터 복구 시 **허용 가능한 데이터 손실 시점** |
| **DRP** (Disaster Recovery Plan) | 장기간 시설 운영이 불가능한 경우를 대비한 **재난 복구 계획** |
| **DRS** (Disaster Recovery System) | 재해복구계획의 수행을 지원하기 위한 인적·물적 자원 및 관리 체계 **재해복구센터** |

---

## 17. AJAX(Asynchronous JavaScript and XML)

> 비동기 통신 기법

- 브라우저의 **XMLHttpRequest 객체**를 이용해 **전체 페이지를 새로 고치지 않고** 일부 데이터만 로드하는 기법
- HTML만으로 구현하기 어려운 다양한 작업을 웹 페이지에서 구현하여 **사용자와 자유롭게 상호작용**할 수 있도록 하는 기술

## 18. 소프트웨어 개발방법론 종류

| 방법론 | 설명 |
|--------|------|
| **구조적 방법론** | 기능에 따라 시스템을 나누어 개발 후 통합하는 **분할과 정복** 방식 |
| **정보공학 방법론** | 정보시스템 개발에 필요한 **관리 절차와 작업 기반을 체계화** |
| **객체지향 방법론** | **'객체'** 라는 기본 단위로 시스템을 분석 및 설계 |
| **컴포넌트 기반 방법론 (CBD)** | 소프트웨어 **컴포넌트를 조립**하여 새로운 응용 프로그램 작성 |
| **애자일 방법론** | 사람 중심, 변화에 **유연하고 신속하게 적응**하는 경량 개발 방법론 |
| **제품 계열 방법론** | 특정 제품에 적용할 **공통 기능을 정의**하여 개발 |

## 19. IPSec 주요 프로토콜

> IPSec: IP 계층(3계층)에서 **무결성·인증·기밀성**을 보장하는 IP 보안 프로토콜

| 프로토콜 | 설명 |
|----------|------|
| **AH** (Authentication Header) | MAC을 이용한 인증 및 송신처 인증 제공 / **기밀성(암호화) 제공 안 함** |
| **ESP** (Encapsulating Security Payload) | MAC + 암호화를 이용한 인증, 송신처 인증, **기밀성 모두 제공** |
| **IKE** (Internet Key Exchange) | Key를 주고받는 알고리즘 / 공개 네트워크에서 Key 교환 방법 정의 |

## 20. 디자인 패턴 - 행위 패턴

> 디자인 패턴: 자주 발생하는 문제들을 재사용할 수 있는 해결책  
> 종류: **생성 패턴** (객체 생성) / **구조 패턴** (클래스·객체 조합) / **행위 패턴** (알고리즘·책임 분배)

| 패턴 | 설명 |
|------|------|
| **Mediator** (중재자) | 객체 수가 많아 통신이 복잡해질 때 **중재자**를 두어 통신 빈도를 줄임 |
| **Interpreter** (해석자) | 구문을 나누고 분리된 구문의 해석을 맡는 클래스를 각각 작성 |
| **Iterator** (반복자) | 구현 방법을 노출하지 않으면서 집합체의 모든 항목에 접근하는 방법 제공 |
| **Template Method** | 일부를 서브 클래스로 캡슐화하여 전체 구조는 유지하면서 특정 단계를 변경 |
| **Observer** (관찰자) | 한 객체 상태 변경 시 의존하는 **다른 객체들에 자동으로 갱신** |
| **State** (상태) | 객체 상태를 캡슐화하여 클래스화하고 이를 참조하는 방식 |
| **Visitor** (방문자) | 처리 기능을 별도 클래스로 분리하여 각 클래스를 돌아다니며 특정 작업 수행 |
| **Command** (명령) | 실행될 기능을 캡슐화하여 **재사용성이 높은 클래스**를 설계 |
| **Strategy** (전략) | 알고리즘 군을 정의하고 각각을 캡슐화하여 **필요 시 교환해서 사용** |
| **Memento** (메멘토) | 객체의 정보를 저장할 필요가 있을 때 적용 |
| **Chain of Responsibility** | 기능 처리의 연결을 **동적으로 변경**할 수 있도록 연결한 패턴 |

## 23. 입력 데이터 검증 및 표현 취약점

| 취약점 | 설명 |
|--------|------|
| **XSS** (Cross Site Script) | 검증되지 않은 외부 입력 데이터가 포함된 웹 페이지를 사용자가 열람 시 **부적절한 스크립트가 실행**되는 공격 |
| **CSRF** (사이트 간 요청 위조) | 사용자가 **자신의 의지와 무관하게** 공격자가 의도한 행위를 특정 웹 사이트에 요청하게 하는 공격 |
| **SQL Injection** | 악의적인 SQL 구문을 삽입하여 **데이터베이스에 접근**하고 정보를 탈취·조작하는 공격 |

## 24. UI 설계 원칙

| 원칙 | 설명 |
|------|------|
| **직관성** (Intuitiveness) | 누구나 쉽게 이해하고 사용할 수 있어야 함 |
| **유효성** (Efficiency) | 정확하고 완벽하게 사용자의 목표가 달성되도록 제작 |
| **학습성** (Learnability) | 초보와 숙련자 모두가 쉽게 배우고 사용할 수 있게 제작 |
| **유연성** (Flexibility) | 사용자의 인터랙션을 최대한 포용하고, 실수를 방지할 수 있도록 제작 |

## 26. 데이터모델링의 절차

> 데이터모델링 구성요소: **개체(Entity) · 속성(Attribute) · 관계(Relation)**

| 단계 | 설명 |
|------|------|
| **개념적 데이터 모델** | 현실 세계에 대한 인식을 **추상적·개념적으로 표현**하여 개념적 구조를 도출 |
| **논리적 데이터 모델** | 업무의 모습을 모델링 표기법으로 **형상화**하여 사람이 이해하기 쉽게 표현 |
| **물리적 데이터 모델** | 논리 데이터 모델을 특정 DBMS의 특성 및 성능을 고려하여 **물리적 스키마로 변환** |

## 27. 형상관리의 절차

> 형상 관리: 소프트웨어 개발 전 과정에서 발생하는 **모든 항목의 변경 사항을 관리**하기 위한 활동

| 절차 | 설명 |
|------|------|
| **형상 식별** | 형상 관리 대상을 정의 및 식별하는 활동 |
| **형상 통제** | 형상 항목의 버전 관리를 위한 **형상통제위원회** 운영 |
| **형상 감사** | 소프트웨어 베이스라인의 **무결성 평가** |
| **형상 기록** | 소프트웨어 형상 및 변경 관리에 대한 각종 수행 결과를 기록 |
