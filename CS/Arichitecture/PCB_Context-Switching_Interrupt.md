# 🖥️  PCB, Context-Switching & Interrupt

## 1. 프로세스의 생성과 PCB

### 🔄 프로그램 실행 흐름
1.  **프로그램 실행 전**: 보조기억장치에 존재하는 **정적 상태의 프로그램**.
2.  **프로그램 실행 직후**: 프로그램이 **메모리(RAM)**에 적재되며 프로세스 주소 공간(`Code`, `Data`, `Stack`)이 생성됨. (이때 CPU는 메모리에 올라간 명령어를 읽어 처리하는 주체이며, 프로그램이 CPU에 직접 올라가는 것이 아님에 유의)
3.  **프로그램 실행 후**: 해당 프로세스의 제어 정보를 담은 메타데이터들이 **PCB**에 저장됨.

### 📋 PCB (Process Control Block)란?
프로세스의 정보를 저장하는 커널 내의 자료구조

*   **포함 요소**:
    *   **PID**: 프로세스 고유 식별 번호.
    *   **프로세스 상태**: 준비(Ready), 실행(Running), 대기(Waiting) 등.
    *   **프로세스 우선순위**: 스케줄링 시 사용되는 순위 정보.
    *   **CPU Registers**: 프로세스가 중단될 때의 레지스터 상태를 저장하는 공간.
    *   **PC (Program Counter)**: 다음으로 실행할 명령어의 주소.
    *   **Owner**: 계정 정보, CPU 사용 시간 등 각종 스케줄러에 필요한 정보.
    *   **기억장치 관리 정보**: 페이지 테이블 정보 등.
    *   **입출력 정보**: 할당된 I/O 장치 및 열린 파일 목록.
    *   **Memory Limit**: 프로세스가 사용 가능한 메모리 경계값.

### 🔗 PCB 관리 방식 (Linked List)
PCB는 **Linked List** 구조로 관리.
*   **특징**: 노드의 삽입과 삭제가 편리하여 프로세스의 생성과 종료, 이동을 빠르고 유연하게 처리.
*   **동작**: 프로세스 종료 시 관련 PCB도 리스트에서 제거.

## 2. 문맥 교환 (Context-Switching)

### 💡 정의 및 필요성
멀티태스킹을 위해 실행 중인 프로세스의 상태를 PCB에 보관하고, 새로 실행할 프로세스의 PCB 정보를 레지스터에 적재하는 과정.

*   **레지스터(Register)**: CPU 전체를 의미하는 것이 아니라, **CPU 내부에 있는 매우 작고 빠른 임시 저장 공간**입니다. CPU가 계산을 수행할 때 사용하는 핵심 장치.
*   **핵심**: 정확히 말하면 **CPU의 레지스터 정보가 변경되는 것**이 문맥 교환의 실체.
*   **Context**: CPU가 다루는 Task(Process/Thread)에 대한 정보이며, 주로 Register에 저장되고 PCB로 관리됨.

### 🚨 발생 기준
1.  **Interrupt 발생**: 하드웨어 신호나 예외 상황.
2.  **시간 할당량 소진**: 프로세스의 선점 허용 기간(Time Quantum) 만료.
3.  **I/O 대기**: 입출력 작업을 위해 프로세스가 대기 상태로 전환될 때.

## 3. 인터럽트(Interrupt)와의 관계

> **⚠️ 주의: 방향성**

*   **인터럽트(Interrupt)**: CPU가 작업을 수행하는 중에 예외적인 사건(I/O 완료, 타이머 만료 등)이 발생했음을 알리는 신호.
*   **흐름**: 인터럽트 신호 수신 → 현재 작업 중단 → 인터럽트 핸들러(ISR) 실행 → **(필요 시) 문맥 교환 발생**.

| 케이스 | 가능 여부 | 설명 |
| :--- | :---: | :--- |
| **Interrupt → Context-Switching** | **O** | 인터럽트 처리 후 스케줄링에 의해 교환 발생 가능 |
| **Context-Switching → Interrupt** | **X** | 문맥 교환이 일어난다고 항상 인터럽트가 발생하는 것은 아님 |

## 4. 오버헤드(Overhead)와 비용

문맥 교환 시 CPU가 실제 연산을 하지 못하는 시간을 **오버헤드**

*   **소모 비용**: Cache 초기화, Memory Mapping 초기화(TLB Flush), 커널의 상시 실행 필요성 등.
*   **시간 할당량의 영향**:
    *   **할당량이 작으면**: 문맥 교환 수와 오버헤드가 증가하지만, 사용자 입장에서는 여러 프로그램이 동시에 잘 돌아가는 것처럼 보임.
    *   **할당량이 크면**: 오버헤드는 감소하지만, 멀티태스킹의 체감이 떨어집니다.

## 5. 프로세스 vs 스레드 문맥 교환

스레드 간 문맥 교환 비용이 프로세스 간 교환보다 훨씬 저렴함.

### 🧵 스레드 문맥 교환이 효율적인 이유
*   **자원 공유**: 프로세스는 `Code`, `Data`, `Heap`, `Stack`을 독립적으로 가지지만, 같은 프로세스 내 스레드들은 **Stack 영역만 독립적이고 나머지는 공유**
*   **교체 범위**: 스레드 교환 시에는 **Stack 포인터와 일부 레지스터 값만 교체**
*   **프로세스 교환의 부담**: 프로세스 간 교환은 메모리 주소 공간 전체가 바뀌므로 **TLB(Translation Lookaside Buffer) 플러시, 캐시 무효화** 등이 발생하여 비용이 매우 큼
