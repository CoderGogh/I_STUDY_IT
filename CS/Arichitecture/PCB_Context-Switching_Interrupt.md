## 1. 프로세스의 생성과 PCB

### 🔄 프로그램 실행 흐름
| 단계 | 상태 | 주요 변화 |
| :--- | :--- | :--- |
| **1. 실행 전** | 정적 프로그램 | 보조기억장치(HDD/SSD)에 파일 형태로 존재함 |
| **2. 실행 직후** | 프로세스 생성 | **메모리(RAM)** 적재 및 주소 공간(`Code`, `Data`, `Stack`) 생성 |
| **3. 실행 중** | 관리 상태 | 커널 영역에 해당 프로세스의 메타데이터인 **PCB** 생성 |

> ⚠️ **주의**: 프로그램은 메모리에 적재되는 것이며, CPU는 메모리의 명령어를 읽어 처리하는 주체일 뿐 프로그램이 CPU에 직접 올라가는 것이 아님

## 📋 PCB (Process Control Block) 구성 요소
운영체제가 프로세스를 제어하기 위해 필요한 정보를 담는 자료구조

*   **PID**: 프로세스 고유 식별 번호
*   **프로세스 상태**: 준비(Ready), 실행(Running), 대기(Waiting) 등
*   **프로세스 우선순위**: 스케줄링 시 사용되는 순위
*   **CPU Registers**: 프로세스 중단 시의 레지스터 상태 저장 공간
*   **PC (Program Counter)**: 다음 실행할 명령어의 주소
*   **Owner**: 계정 정보, CPU 사용 시간 등 스케줄러용 정보
*   **기억장치 관리**: 페이지 테이블 정보 등
*   **입출력 정보**: 할당된 I/O 장치 및 열린 파일 목록
*   **Memory Limit**: 프로세스가 사용 가능한 메모리 경계값

**📋 주요 프로세스 상태 코드**
운영체제의 프로세스 상태 전이 모델이 실제 터미널에서는 아래와 같은 코드로 표현됩니다.

| 코드 | 상태 | 의미 |
| :--- | :--- | :--- |
| **R** | **Running / Runnable** | CPU를 점유하여 **실행 중**이거나, 할당받기 위해 **준비(Ready) 상태**로 대기 중임. |

| **Z** | **Zombie** | 실행은 끝났지만 부모가 확인하지 않아 **PCB가 메모리에 남은** 상태. |
| **I** | **Idle** | 커널 스레드가 할 일이 없어 쉬고 있는 유휴 상태. (macOS 등에서 확인) |

**🔍 추가 기호 (Suffix)**
코드 뒤에 붙는 소문자는 해당 프로세스의 세부적인 특성을 나타냄.

*   **`<`**: **높은 우선순위**를 가진 프로세스.
*   **`N`**: **낮은 우선순위**를 가진 프로세스.
*   **`s`**: **세션 리더** (자식 프로세스들을 관리하는 제어 프로세스).
*   **`+`**: 현재 사용자의 터미널에서 실행 중인 **포그라운드(Foreground)** 프로세스.

### 🔗 PCB 관리 방식: Linked List
*   **특징**: 노드의 삽입/삭제가 편리하여 프로세스의 생성과 종료를 빠르고 유연하게 처리함.
*   **동작**: 프로세스가 종료되면 해당 PCB는 리스트에서 즉시 제거됨.


### 🔗 사용자 모드 vs 커널 모드 (User & Kernel Mode)

CPU는 실행 권한을 구분하여 시스템 자원을 보호함.

* **사용자 모드 (User Mode)**: 애플리케이션 코드가 실행되는 모드로, 하드웨어 및 핵심 메모리 접근이 제한됨.
* **커널 모드 (Kernel Mode)**: OS의 커널이 실행되는 모드입니다. 모든 CPU 명령어를 실행할 수 있으며 하드웨어를 직접 제어할 수 있는 완전한 권한을 가짐.
* **전환 과정**: 애플리케이션이 파일 읽기 등 특권 작업이 필요할 때 **시스템 콜(System Call)**을 호출하면 CPU는 커널 모드로 전환되어 작업을 처리함.


### 📊 PCB vs TCB 상세 비교

| 구분 | PCB (Process Control Block) | TCB (Thread Control Block) |
| :--- | :--- | :--- |
| **개념** | 운영체제가 프로세스를 제어하기 위한 정보 저장소 | 스레드 단위로 실행 상태를 관리하기 위한 저장소 |
| **소유 및 생성** | 프로세스당 1개 생성 | 스레드당 1개 생성 |
| **관리 범위** | **가족 전체의 자원**: 메모리 주소 공간(Page Table), 열린 파일, I/O 상태, PID 등 | **개별 구성원의 흐름**: Register 세트, PC(Program Counter), Stack Pointer 등 |
| **메모리 관계** | 독자적인 가상 메모리 공간 정보를 가짐 | **부모 프로세스의 PCB 자원(Code, Data, Heap)을 공유**함 |
| **스위칭 비용** | **매우 높음**: 주소 공간 전환 및 TLB 캐시 플러시 발생 | **낮음**: 주소 공간 유지, 레지스터 정보만 교체 |
| **상호 관계** | 내부에 해당 프로세스의 TCB 리스트를 포함/관리함 | 자신이 속한 부모 PCB에 대한 포인터를 가짐 |

## 🧠 MMU와 TLB 핵심 요약

### 1. MMU (Memory Management Unit)
CPU와 RAM 사이에서 주소를 변환하고 관리하는 하드웨어 장치입니다.

*   **주소 변환**: CPU의 **가상 주소**를 실제 RAM의 **물리 주소**로 실시간 변환합니다.
*   **메모리 보호**: 페이지 테이블의 권한 비트를 체크하여 잘못된 메모리 접근(읽기/쓰기 권한 위반 등)을 차단합니다.
*   **단편화 해결**: 물리적으로 흩어진 메모리를 가상 공간에서 연속적인 형태로 매핑하여 관리 효율을 높입니다.



---

### 2. TLB (Translation Lookaside Buffer)
MMU 내부에 위치한 **주소 변환 전용 고속 캐시 메모리**입니다.

*   **성능 최적화**: 메인 메모리의 페이지 테이블에 직접 접근하는 시간을 줄이기 위해 자주 참조되는 변환 정보를 저장합니다.
*   **TLB Hit**: 요청한 주소 정보가 TLB에 있어 즉시 물리 주소로 변환되는 상태 (매우 빠름).
*   **TLB Miss**: 정보가 없어 RAM의 페이지 테이블을 직접 조회해야 하는 상태 (상대적으로 느림).



---

### 📊 MMU vs TLB 비교

| 구분 | MMU | TLB |
| :--- | :--- | :--- |
| **정체** | 주소 변환 관리 하드웨어 유닛 | 주소 변환 정보 저장용 고속 캐시 |
| **위치** | CPU와 메모리 사이 | MMU 내부 |
| **핵심 역할** | 가상 메모리 체계 총괄 및 보호 | 주소 변환 지연 시간(Latency) 단축 |
| **특이사항** | 프로세스 스위칭 시 제어 주체 | 프로세스 스위칭 시 **Flush(초기화)** 발생 |

## 2. 문맥 교환 (Context-Switching)

### 💡 정의 및 핵심 개념
멀티태스킹을 위해 현재 실행 중인 프로세스의 상태를 PCB에 보관하고, 새 프로세스의 정보를 레지스터에 적재하는 과정.

*   **레지스터(Register)**: CPU 내부에 있는 매우 작고 빠른 임시 저장 공간으로, 연산의 핵심 장치임.
*   **핵심 실체**: **CPU 레지스터 정보가 변경되는 것**이 문맥 교환의 본질임.
*   **Context**: CPU가 Task를 실행하기 위해 필요한 정보(Register, PC 등)를 의미하며 PCB에서 관리됨.


### 🚨 발생 기준 (Trigger)
1.  **Interrupt 발생**: 하드웨어 신호나 예외 상황 발생 시.
2.  **시간 할당량(Time Quantum) 소진**: 프로세스의 선점 허용 기간 만료 시.
3.  **I/O 대기**: 입출력 작업을 위해 프로세스가 대기(Blocked) 상태로 전환될 때.

## 3. 인터럽트(Interrupt)와 처리 과정

### ⚡ 인터럽트 방식 vs 폴링(Polling) 방식
컨트롤러가 입력을 받아들이는 방법에는 두 가지가 있음.

| 방식 | 감지 주체 | 특징 | 장단점 |
| :--- | :--- | :--- | :--- |
| **폴링(Polling)** | 소프트웨어 | 사용자가 명령어로 입력 핀 값을 계속 읽어 변화를 확인 | 하드웨어 대비 속도가 느리고 CPU가 원래 일에 집중하기 어려움 |
| **인터럽트(Interrupt)** | 하드웨어 | MCU 자체가 하드웨어적으로 변화를 체크하여 이벤트 발생 시에만 동작 | 신속한 대응이 가능하며 실시간 대응이 필요한 경우 필수적임 |

## ⚡ 인터럽트의 분류와 특징

---

### 1. 인터럽트의 세 가지 종류

#### ① 하드웨어 인터럽트 (External Interrupt)
가장 일반적인 형태의 인터럽트로, CPU 외부의 하드웨어 장치가 CPU에게 보내는 물리적 신호입니다.
* **예시**: 키보드/마우스 입력, 디스크 읽기 완료, 네트워크 패킷 수신 등.
* **특징**: 발생 시점을 예측할 수 없는 **비동기적(Asynchronous)** 성격을 가집니다.

#### ② 소프트웨어 인터럽트 (Trap / System Call)
프로그램 실행 중 명령어를 통해 **의도적으로** 운영체제에게 서비스를 요청하는 인터럽트입니다.
* **예시**: 파일을 읽기 위한 `read()` 시스템 콜, 프로세스 생성을 위한 `fork()` 등.
* **특징**: 프로그램 코드 내 특정 지점에서 발생하므로 **동기적(Synchronous)**입니다.

#### ③ 예외 인터럽트 (Exception / Fault)
CPU가 명령어를 실행하는 도중 예기치 못한 **오류나 특수한 상황**이 발생했을 때 나타납니다.
* **예시**: 0으로 나누기(Divide by Zero), 권한 외 메모리 접근(Segmentation Fault), 페이지 부재(Page Fault).
* **특징**: 프로그램 실행 중 예기치 않게 발생하며, 소프트웨어 인터럽트처럼 **동기적(Synchronous)**입니다.

---

### 2. 공통점과 차이점

#### 🤝 공통점
1.  **커널 모드 진입**: 인터럽트 발생 시 CPU는 즉시 **커널 모드**로 전환되어 관리 권한을 얻습니다.
2.  **Context Saving**: 현재 작업을 중단하고 레지스터, PC 등 현재 상태를 **스택(Stack)**이나 **PCB**에 저장합니다.
3.  **ISR 실행**: 인터럽트 벡터 테이블을 참조하여 해당 번호에 맞는 **ISR(인터럽트 서비스 루틴)**을 호출합니다.

#### 🔍 차이점 비교 테이블

| 구분 | 하드웨어 인터럽트 | 소프트웨어 인터럽트 | 예외 (Exception) |
| :--- | :--- | :--- | :--- |
| **발생 원인** | 외부 하드웨어 장치 신호 | 프로그램 내부 시스템 콜 | 명령어 실행 중 오류/예외 |
| **발생 시점** | **비동기적** (예측 불가) | **동기적** (코드 위치 고정) | **동기적** (오류 발생 시) |
| **의도성** | 비의도적 (외부 이벤트) | **의도적** (자원 요청) | 비의도적 (프로그램 오류) |
| **대응 방식** | 서비스 처리 후 복귀 | 서비스 수행 후 복귀 | 오류 수정 또는 프로그램 종료 |

1.  **중단**: 주 프로그램 실행 중 인터럽트 신호 수신 시 현재 작업을 멈춤.

> **핵심**: 인터럽트는 발생 시기를 예측하기 힘든 경우에 컨트롤러가 가장 빠르게 대응할 수 있는 방법임.
> **예시**: pooling은 작업의 완료 상황을 모른 채 CPU가 주기적으로 확인(리소스 낭비)
> **예시**: Interrupt는 CPU가 일을 하다가 외부 장치에서 작업이 끝났다는 이벤트(신호)를 받고서 확인

| **Context-Switching → Interrupt** | **X** | 문맥 교환 자체가 인터럽트를 유발하는 것은 아님 |

## 4. 오버헤드(Overhead)와 비용

문맥 교환 중 CPU가 실제 연산을 하지 못하고 소모되는 시간을 **오버헤드**

### 💸 주요 비용 (Cost) 요소
*   **Cache 초기화**: 기존 데이터가 무효화되어 성능 일시 저하.
*   **Memory Mapping 초기화**: TLB(Translation Lookaside Buffer) 플러시 발생.
*   **Kernel 개입**: 주소 공간 전환 등을 위해 커널 모드가 상시 실행되어야 함.

### ⏳ 시간 할당량(Time Quantum)의 영향
| 할당량 크기 | 오버헤드 | 사용자 체감 (응답성) |
| :--- | :---: | :--- |
| **작을 때** | **증가** | 여러 프로그램이 동시에 매끄럽게 돌아가는 듯함 (Good) |
| **클 때** | **감소** | 멀티태스킹 체감이 떨어지고 응답이 둔해짐 (Bad) |

## 5. 프로세스 vs 스레드 문맥 교환

<details>
<summary><b>🧵 Thread 와 Process 중 Context Switching Cost가 높은 것은? (클릭하여 확인)</b></summary>

### 💡 핵심 요약
스레드 간의 문맥 교환은 프로세스 간 교환보다 비용이 훨씬 저렴합니다.

### 효율적인 이유
1.  **자원 공유**: 스레드들은 프로세스의 `Code`, `Data`, `Heap` 영역을 공유함.
2.  **교체 범위 최소화**: 스레드 교환 시에는 **Stack 포인터와 일부 레지스터 값만 교체**하면 됨.
3.  **캐시 유지**: 프로세스 교환과 달리 메모 주소 공간이 바뀌지 않으므로 **TLB 플러시나 캐시 무효화** 부담이 거의 없음.


### 차이가 발생하는 이유
1.  **자원 공유**: 같은 프로세스 내의 스레드들은 `Code`, `Data`, `Heap` 영역을 공유합니다. 따라서 메모리 주소 공간을 통째로 바꿀 필요가 없습니다.
2.  **교체 범위 최소화**: 스레드 스위칭은 TCB(Thread Control Block)에 담긴 **스택 포인터와 레지스터 값**만 갈아 끼우면 끝납니다.
3.  **TLB 성능 유지 (결정적 차이)**: 
    * **프로세스 스위칭**: 가상 메모리 주소 체계가 바뀌므로 주소 변환 정보를 담은 **TLB(Translation Lookaside Buffer) 캐시를 완전히 비워야(Flush)** 합니다.
    * **스레드 스위칭**: 동일한 주소 공간을 사용하므로 **TLB를 비울 필요가 없어** 전환 후에도 메모리 접근 속도가 유지됩니다.

</details>
